# Roadmap Migliorie aReS Core

## Obiettivo
- Consolidare `@ares/core` come libreria framework con API stabili, comportamento prevedibile e dipendenze applicative esplicite.

## v1.0.x Stabilizzazione Core
- [ ] Correggere i bug concreti in `xhr.js`, `scripts.js`, `objects.js`, `permissions.js` e `geographical.js`.
- [ ] Rimuovere o confinare gli override globali di `console.js`, sostituendoli con logger o adapter opzionali.
- [ ] Sistemare `errorHandling.js` e uniformare il contratto minimo di successo/errore nelle utility di base.
- [ ] Ridurre i bug logici in `dataDescriptors.js` e nei punti piu' fragili di `datasources.js`.
- [ ] Introdurre smoke test minimi per bootstrap, scripting, networking e datasource helpers.

## v1.1 Contratto Pubblico
- [ ] Definire la superficie pubblica di `@ares/core`: entrypoint, moduli supportati e livelli di stabilita'.
- [ ] Distinguere chiaramente utility interne, helper sperimentali e API pensate per i consumer del framework.
- [ ] Rimuovere le dipendenze hard-coded da file esterni al pacchetto, come `permissionData.js`, e sostituirle con configurazione o dependency injection.
- [ ] Standardizzare il formato errori e il comportamento delle utility async e sync.

## v1.2 Configurabilita e Sicurezza
- [ ] Rendere configurabili logging, diagnostica e payload di debug.
- [ ] Rivedere `security.js` per introdurre una strategia di derivazione chiavi e salt non hard-coded.
- [ ] Isolare o ripensare lo scripting dinamico basato su `new Function()`.
- [ ] Esporre opzioni esplicite per moduli con effetti esterni, come networking, geocoding e datasource runtime.

## v1.3 Architettura e Packaging
- [ ] Spezzare il core in sottopacchetti o entrypoint secondari dove il coupling e' troppo alto.
- [ ] Aggiungere `exports` in `package.json` e chiarire gli import supportati dopo publish.
- [ ] Valutare se componenti come CLI, geocoding e datasource runtime debbano stare nel pacchetto base o in moduli dedicati.
- [ ] Allineare il manifest alle dipendenze realmente necessarie nel runtime minimo del core.

## v1.4 Qualita e Adozione
- [ ] Aggiungere test di regressione sulle utility piu' instabili, in particolare `scripts`, `objects`, `permissions` e `xhr`.
- [ ] Aggiungere test di integrazione sui datasource e sui flussi di validazione.
- [ ] Ripulire `README.md` e pubblicare una documentazione realmente presente nel repository.
- [ ] Documentare setup minimo, punti di estensione e compatibilita' attese per chi usa il framework.

## Backlog
- [ ] Valutare una policy di versioning semantico esplicita per i moduli pubblici del core.
- [ ] Introdurre esempi applicativi separati dal pacchetto per mostrare l'integrazione senza mischiare framework e app host.
- [ ] Valutare metriche o tracing opzionali senza imporre effetti collaterali globali al processo Node.
- [ ] Razionalizzare la nomenclatura delle utility e la coerenza tra moduli sincroni, asincroni e classi runtime.
